# 要件定義書

## はじめに

佐賀バスナビゲーターアプリにおいて、方向判定機能はDataLoaderに統合されていますが、時刻表UIに方向情報が表示されていません。この要件定義書では、時刻表UIに方向情報を表示し、ユーザーが路線の往路・復路を視覚的に理解できるようにするための要件を定義します。

## 用語集

- **System**: 佐賀バスナビゲーターアプリケーション
- **TimetableUI**: 時刻表表示を担当するUIコンポーネント
- **direction**: 路線の方向（'0'=往路、'1'=復路、'unknown'=不明）
- **routeMetadata**: 路線メタデータ（方向判定成功率などの統計情報を含む）
- **directionDetectionRate**: 方向判定成功率（0.0-1.0）
- **検索結果リスト**: バス停間の直通便検索結果を表示するリスト
- **時刻表モーダル**: 路線の詳細時刻表を表示するモーダルウィンドウ

## 要件

### 要件1: 検索結果リストへの方向情報表示

**ユーザーストーリー:** ユーザーとして、検索結果リストで各便の方向（往路/復路）を確認したい。これにより、目的地への正しい方向のバスを選択できる。

#### 受入基準

1. WHEN 検索結果が表示される時 THEN Systemは各便の方向情報を表示する
2. WHEN 方向が'0'（往路）の時 THEN Systemは「往路」または「→」のラベルを表示する
3. WHEN 方向が'1'（復路）の時 THEN Systemは「復路」または「←」のラベルを表示する
4. WHEN 方向が'unknown'の時 THEN Systemは方向ラベルを表示しない、または「方向不明」と表示する
5. WHEN 方向ラベルを表示する時 THEN Systemは視覚的に識別しやすい色とアイコンを使用する

### 要件2: 時刻表モーダルへの方向情報表示

**ユーザーストーリー:** ユーザーとして、時刻表モーダルで各便の方向を確認したい。これにより、時刻表全体の中で往路と復路を区別できる。

#### 受入基準

1. WHEN 時刻表モーダルが表示される時 THEN Systemは各便の方向情報を表示する
2. WHEN 時刻表テーブルに方向列を追加する時 THEN Systemは「方向」列ヘッダーを表示する
3. WHEN 各便の方向を表示する時 THEN Systemは往路/復路のラベルまたはアイコンを表示する
4. WHEN 方向が'unknown'の便がある時 THEN Systemは「-」または空欄を表示する
5. WHEN 時刻表をソートする時 THEN Systemは方向でグループ化するオプションを提供する

### 要件3: 路線メタデータの表示

**ユーザーストーリー:** システム管理者として、路線選択画面で方向判定成功率を確認したい。これにより、データ品質を監視できる。

#### 受入基準

1. WHEN 路線選択画面が表示される時 THEN Systemは各路線の方向判定成功率を表示する
2. WHEN 方向判定成功率が50%未満の時 THEN Systemは警告バッジを表示する
3. WHEN 方向判定成功率が50%以上80%未満の時 THEN Systemは注意バッジを表示する
4. WHEN 方向判定成功率が80%以上の時 THEN Systemはバッジを表示しない、または成功バッジを表示する
5. WHEN バッジをクリックする時 THEN Systemは方向判定の詳細情報をツールチップで表示する

### 要件4: 方向フィルタリング機能

**ユーザーストーリー:** ユーザーとして、時刻表モーダルで往路または復路のみを表示したい。これにより、必要な方向の便だけを確認できる。

#### 受入基準

1. WHEN 時刻表モーダルが表示される時 THEN Systemは方向フィルタボタンを表示する
2. WHEN 「往路のみ」ボタンをクリックする時 THEN Systemは往路（direction='0'）の便のみを表示する
3. WHEN 「復路のみ」ボタンをクリックする時 THEN Systemは復路（direction='1'）の便のみを表示する
4. WHEN 「すべて」ボタンをクリックする時 THEN Systemは全ての便を表示する
5. WHEN フィルタを適用する時 THEN Systemは現在のフィルタ状態をボタンの視覚的スタイルで示す

### 要件5: 方向情報のアクセシビリティ

**ユーザーストーリー:** 視覚障害のあるユーザーとして、スクリーンリーダーで方向情報を理解したい。これにより、全てのユーザーが方向情報にアクセスできる。

#### 受入基準

1. WHEN 方向ラベルを表示する時 THEN Systemは適切なaria-label属性を設定する
2. WHEN 方向アイコンを使用する時 THEN Systemはアイコンに代替テキストを提供する
3. WHEN 方向フィルタボタンを表示する時 THEN Systemはaria-pressed属性で選択状態を示す
4. WHEN 方向判定バッジを表示する時 THEN Systemはaria-describedby属性でツールチップを関連付ける
5. WHEN キーボードナビゲーションを使用する時 THEN Systemは全ての方向関連UIにフォーカス可能にする

### 要件6: レスポンシブデザイン対応

**ユーザーストーリー:** モバイルユーザーとして、小さい画面でも方向情報を確認したい。これにより、全てのデバイスで方向情報が利用できる。

#### 受入基準

1. WHEN モバイル画面で表示する時 THEN Systemは方向ラベルをアイコンのみに省略する
2. WHEN タブレット画面で表示する時 THEN Systemは方向ラベルを短縮形（「往」「復」）で表示する
3. WHEN デスクトップ画面で表示する時 THEN Systemは方向ラベルを完全形（「往路」「復路」）で表示する
4. WHEN 画面幅が狭い時 THEN Systemは方向列の幅を最小限に抑える
5. WHEN タッチデバイスで使用する時 THEN Systemは方向フィルタボタンのタップ領域を十分に確保する

### 要件7: パフォーマンス最適化

**ユーザーストーリー:** 開発者として、方向情報の表示がアプリケーションのパフォーマンスに影響しないようにしたい。これにより、ユーザー体験が損なわれない。

#### 受入基準

1. WHEN 検索結果を表示する時 THEN Systemは方向情報の取得に10ms以上かからない
2. WHEN 時刻表モーダルを表示する時 THEN Systemは方向情報の表示に50ms以上かからない
3. WHEN 方向フィルタを適用する時 THEN Systemはフィルタリングに100ms以上かからない
4. WHEN 大量の便データを表示する時 THEN Systemは仮想スクロールを使用してレンダリングを最適化する
5. WHEN 方向情報をキャッシュする時 THEN Systemはメモリ使用量を最小限に抑える

### 要件8: エラーハンドリング

**ユーザーストーリー:** 開発者として、方向情報が取得できない場合の適切な処理を実装したい。これにより、アプリケーションが安定して動作する。

#### 受入基準

1. WHEN 方向情報が存在しない時 THEN Systemはデフォルト値（'unknown'）を使用する
2. WHEN 方向情報の取得に失敗する時 THEN Systemはエラーログを出力し、処理を継続する
3. WHEN 方向判定成功率が計算できない時 THEN Systemは「N/A」と表示する
4. WHEN 方向フィルタリング中にエラーが発生する時 THEN Systemはフィルタをリセットし、全ての便を表示する
5. WHEN 方向情報の表示中にエラーが発生する時 THEN Systemはユーザーに通知せず、ログに記録する

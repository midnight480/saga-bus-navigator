# 要件定義書

## はじめに

佐賀市バスナビゲーションアプリに、OpenRouteService (ORS) APIを使用した道路に沿ったバス経路描画機能を追加します。現在は直線で描画されているバス路線を、実際の道路ネットワークに沿った経路として表示することで、ユーザーにより正確な経路情報を提供します。

## 用語集

- **ORS**: OpenRouteService - オープンソースの経路検索サービス
- **経路描画器**: バス経路を地図上に描画するコンポーネント
- **APIクライアント**: ORS APIと通信するクライアント
- **キャッシュ管理器**: API応答をキャッシュして管理するコンポーネント
- **座標**: 座標情報（緯度・経度）
- **GeoJSON**: 地理空間データを表現するJSON形式
- **バス停**: バス停の位置情報と属性を持つエンティティ
- **経路区間**: 2つのバス停間の経路区間

## 要件

### 要件1: ORS API統合

**ユーザーストーリー:** 開発者として、ORS Directions APIを統合したい。そうすることで、バス停間の道路に沿った経路を取得できる。

#### 受入基準

1. WHEN APIクライアントがORS Directions APIにリクエストを送信する THEN システムはAuthorizationヘッダーにAPIキーを含めなければならない
2. WHEN APIクライアントが座標を準備する THEN システムは座標を[経度, 緯度]のペアとしてフォーマットしなければならない
3. WHEN APIクライアントがレスポンスを受信する THEN システムはレスポンスからGeoJSON geometryを解析しなければならない
4. IF APIがエラーを返す THEN システムはエラーをログに記録し、失敗ステータスを返さなければならない
5. WHEN APIクライアントがリクエストボディを構築する THEN システムは"coordinates"フィールドに座標ペアの配列を含めなければならない

### 要件2: 経路描画

**ユーザーストーリー:** ユーザーとして、実際の道路に沿ったバス経路を見たい。そうすることで、バスが通る実際の経路を理解できる。

#### 受入基準

1. WHEN 経路GeoJSONを受信する THEN 経路描画器はLeaflet地図上に表示しなければならない
2. WHEN 複数の経路区間が存在する THEN 経路描画器は一貫したスタイルで全ての区間を描画しなければならない
3. WHEN 経路を描画する THEN システムはバス経路に視覚的に区別できる色を使用しなければならない
4. WHEN 経路を地図に追加する THEN 経路描画器はLeafletレイヤーへの参照を保存しなければならない
5. WHEN ユーザーが経路のクリアを要求する THEN 経路描画器は地図から全ての経路レイヤーを削除しなければならない

### 要件3: API制限管理

**ユーザーストーリー:** システム管理者として、ORS APIのレート制限を遵守したい。そうすることで、アプリケーションが無料プランの制約内で動作し続けられる。

#### 受入基準

1. THE システムはDirections APIの呼び出しを1日あたり最大2000リクエストに制限しなければならない
2. THE システムはDirections APIの呼び出しを1分あたり最大40リクエストに制限しなければならない
3. WHEN レート制限に近づく THEN システムは重要でないリクエストをキューに入れるか延期しなければならない
4. WHEN レート制限を超える THEN システムはキャッシュされた結果を返すかエラーメッセージを表示しなければならない
5. THE システムは現在の時間枠内で行われたAPI呼び出しの数を追跡しなければならない

### 要件4: キャッシュ機構

**ユーザーストーリー:** 開発者として、経路レスポンスをキャッシュしたい。そうすることで、API呼び出しを最小化し、パフォーマンスを向上できる。

#### 受入基準

1. WHEN 経路が正常に取得される THEN キャッシュ管理器は座標配列をキーとしてGeoJSONレスポンスを保存しなければならない
2. WHEN 経路リクエストが行われる THEN キャッシュ管理器はAPIを呼び出す前にキャッシュされたレスポンスが存在するか確認しなければならない
3. WHEN キャッシュされた経路が見つかる THEN システムはAPI呼び出しを行わずにキャッシュされたGeoJSONを使用しなければならない
4. THE キャッシュ管理器は永続化のためにブラウザのlocalStorageまたはsessionStorageを使用しなければならない
5. WHEN キャッシュストレージが失敗する THEN システムはエラーをスローせずにAPI呼び出しを続行しなければならない

### 要件5: ユーザートリガー実行

**ユーザーストーリー:** ユーザーとして、必要なときだけ経路を描画してほしい。そうすることで、アプリケーションがAPI呼び出しを無駄にしない。

#### 受入基準

1. WHEN ユーザーが地図上で特定のバス路線を選択する THEN システムはその経路を取得して描画しなければならない
2. WHEN ユーザーがバス停の時刻表を表示する THEN システムはそのバス停に停車するバスの経路を描画しなければならない
3. THE システムはアプリケーション起動時に全ての経路を自動的に描画してはならない
4. WHEN ユーザーが一定レベル以上にズームアウトする THEN システムは詳細な経路描画を非表示にしなければならない
5. WHEN ユーザーが特定のバスを検索する THEN システムは関連する経路のみを描画しなければならない

### 要件6: エラーハンドリング

**ユーザーストーリー:** ユーザーとして、エラーが発生しても適切に処理してほしい。そうすることで、経路検索が失敗しても他の機能を使い続けられる。

#### 受入基準

1. IF ORS APIが利用できない THEN システムはフォールバックとして直線経路を表示しなければならない
2. WHEN ネットワークエラーが発生する THEN システムは指数バックオフで最大3回リクエストを再試行しなければならない
3. IF 全ての再試行が失敗する THEN システムはユーザーにエラーメッセージを表示しなければならない
4. WHEN 無効な座標が提供される THEN システムはAPI呼び出しを行う前にそれらを検証しなければならない
5. IF APIが429(レート制限)エラーを返す THEN システムは一時的に利用できないことを示すメッセージを表示しなければならない

### 要件7: 座標変換

**ユーザーストーリー:** 開発者として、異なる形式間で座標を正しく変換したい。そうすることで、ORS APIが有効な入力を受け取れる。

#### 受入基準

1. WHEN バス停座標が[緯度, 経度]形式である THEN システムはORS API用に[経度, 緯度]に変換しなければならない
2. WHEN ORSがGeoJSON座標を返す THEN システムはLeaflet表示用に正しく処理しなければならない
3. THE システムは緯度の値が-90から90の間であることを検証しなければならない
4. THE システムは経度の値が-180から180の間であることを検証しなければならない
5. IF 座標検証が失敗する THEN システムはAPIを呼び出す前にリクエストを拒否しなければならない

### 要件8: 既存機能との統合

**ユーザーストーリー:** ユーザーとして、新しい経路機能が既存の地図機能とシームレスに動作してほしい。そうすることで、一貫した体験が得られる。

#### 受入基準

1. WHEN 経路を描画する THEN システムは地図上の既存のバス停マーカーを維持しなければならない
2. WHEN ユーザーがバス停マーカーをクリックする THEN システムは既存のポップアップ情報を引き続き表示しなければならない
3. THE 経路描画器は新しいものを作成せずに既存のLeaflet地図インスタンスを使用しなければならない
4. WHEN 経路を描画する THEN システムは明示的に境界に合わせるよう要求されない限り、現在の地図のズームと中心を保持しなければならない
5. THE システムはベースレイヤーとしてOpenStreetMapタイルを引き続き使用しなければならない

### 要件9: パフォーマンス最適化

**ユーザーストーリー:** ユーザーとして、経路描画が高速で応答性が高いことを望む。そうすることで、アプリケーションがスムーズに感じられる。

#### 受入基準

1. WHEN 複数の経路区間を描画する必要がある THEN システムは可能な限りAPIリクエストをバッチ処理しなければならない
2. WHEN 経路が既に表示されている THEN システムは不必要に再描画してはならない
3. THE システムは過度なAPI呼び出しを防ぐために、連続する経路リクエストをデバウンスしなければならない
4. WHEN 複雑な経路を描画する THEN システムはLeafletの効率的なレンダリングメソッドを使用しなければならない
5. THE システムはORS APIに送信する座標点の最大数をリクエストあたり50に制限しなければならない

### 要件10: 設定管理

**ユーザーストーリー:** 開発者として、ORS API設定を一元管理したい。そうすることで、設定を簡単に更新できる。

#### 受入基準

1. THE システムはORS APIキーを設定オブジェクトに保存しなければならない
2. THE システムはORS APIベースURLを設定オブジェクトに保存しなければならない
3. THE システムはルーティングプロファイル(例: driving-car, foot-walking)の設定を許可しなければならない
4. THE システムはキャッシュ有効期限の設定を許可しなければならない
5. WHERE 設定値が欠落している THEN システムは妥当なデフォルト値を使用しなければならない

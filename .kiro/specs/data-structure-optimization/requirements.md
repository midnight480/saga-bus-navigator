# 要件定義書

## はじめに

佐賀バスナビゲーターアプリにおいて、双方向路線対応の基本実装は完了していますが、データ構造とインデックス戦略に改善の余地があります。現在のフラットなデータ構造では、方向を考慮した経路ナビゲーション、効率的な検索、停留所の整理が困難です。この要件定義書では、データ構造の最適化とインデックス戦略の改善を定義します。

## 用語集

- **System**: 佐賀バスナビゲーターアプリケーション
- **GTFS**: General Transit Feed Specification（公共交通データの標準形式）
- **direction_id**: GTFSにおける路線の方向を示すフィールド（0=往路、1=復路）
- **trip**: 特定の時刻に特定の路線を運行する便
- **stop_sequence**: 便における停留所の順序番号
- **trip_headsign**: 便の行き先を示すフィールド
- **parent_station**: 停留所の親駅を示すフィールド
- **逆引きインデックス**: 子から親への参照を高速化するデータ構造
- **方向別インデックス**: 路線と方向の組み合わせでデータを整理した構造

## 要件

### 要件1: 方向判定の改善

**ユーザーストーリー:** システム管理者として、direction_idが空文字列の場合でも正しく方向を判定したい。また、headsignベースの判定が失敗した場合のフォールバック処理を追加したい。

#### 受入基準

1. WHEN tripのdirection_idが空文字列、null、またはundefinedの時 THEN Systemはこれらを全て無効な値として扱う
2. WHEN headsignベースの判定が失敗した時 THEN Systemは停留所順序を分析して方向を推測する
3. WHEN 同じ路線の全tripについて最初と最後の停留所を比較する時 THEN Systemは始点・終点の組み合わせが2パターンあればそれぞれを異なる方向として扱う
4. WHEN 方向判定結果をキャッシュする時 THEN Systemは路線ごとに判定結果を保存し同じ路線の判定を繰り返さない

### 要件2: 方向別時刻表インデックスの追加

**ユーザーストーリー:** 開発者として、路線と方向の組み合わせで時刻表データを高速に検索したい。これにより、往路と復路のスケジュールを簡単に分離できる。

#### 受入基準

1. WHEN DataLoaderがデータを読み込む時 THEN Systemは方向別時刻表インデックスを生成する
2. WHEN 方向別時刻表インデックスを参照する時 THEN Systemは路線IDと方向で時刻表データを取得できる
3. WHEN 時刻表データが変換される時 THEN Systemは既存のtimetable配列から方向別インデックスを生成する
4. WHEN 方向が'unknown'の時刻表エントリが存在する時 THEN Systemはそれらを'unknown'キーの配列に格納する

### 要件3: Trip-Stopマッピングの作成

**ユーザーストーリー:** 開発者として、各tripがどの停留所を順番に経由するかを簡単に取得したい。これにより、経路の可視化や「このtripの全停留所」の表示が容易になる。

#### 受入基準

1. WHEN DataLoaderがデータを読み込む時 THEN SystemはTrip-Stopマッピングを生成する
2. WHEN Trip-Stopマッピングを参照する時 THEN SystemはtripIdから停留所IDの順序付きリストを取得できる
3. WHEN 停留所リストを取得する時 THEN Systemは各停留所について、stopId、stopName、sequence、arrivalTimeを含む
4. WHEN 停留所リストを生成する時 THEN Systemはstop_sequenceでソートされた順序で返す

### 要件4: 路線メタデータの生成

**ユーザーストーリー:** 開発者として、路線レベルの方向情報、行き先一覧、便数を簡単に取得したい。これにより、「この路線は双方向運行か」「この路線の行き先は何か」という質問に簡単に答えられる。

#### 受入基準

1. WHEN DataLoaderがデータを読み込む時 THEN Systemは路線メタデータを生成する
2. WHEN 路線メタデータを参照する時 THEN Systemは路線IDから利用可能な方向のリストを取得できる
3. WHEN 路線メタデータを参照する時 THEN Systemは路線IDから全てのheadsignのリストを取得できる
4. WHEN 路線メタデータを参照する時 THEN Systemは路線IDから方向別のtrip数を取得できる

### 要件5: 逆引きインデックスの追加

**ユーザーストーリー:** 開発者として、「この停留所に停車する全trip」や「この路線+方向の全trip」を効率的に検索したい。

#### 受入基準

1. WHEN DataLoaderがデータを読み込む時 THEN SystemはstopToTripsインデックスを生成する
2. WHEN stopToTripsインデックスを参照する時 THEN Systemは停留所IDから停車する全tripIdのリストを取得できる
3. WHEN DataLoaderがデータを読み込む時 THEN SystemはrouteToTripsインデックスを生成する
4. WHEN routeToTripsインデックスを参照する時 THEN Systemは路線IDと方向から該当する全tripIdのリストを取得できる

### 要件6: 停留所グループ化

**ユーザーストーリー:** ユーザーとして、同じ駅の複数の乗り場を親駅単位でグループ化して表示したい。これにより、検索結果が整理され、混乱を避けられる。

#### 受入基準

1. WHEN DataLoaderがstopsデータを読み込む時 THEN Systemはparent_stationフィールドを保持する
2. WHEN 停留所グループ化インデックスを生成する時 THEN Systemはparent_stationでグループ化した停留所リストを作成する
3. WHEN 停留所を検索する時 THEN Systemは親駅単位でグループ化された結果を返すオプションを提供する
4. WHEN 停留所名に乗り場番号が含まれる時 THEN Systemは重複排除やグループ化ロジックを適用する

### 要件7: パフォーマンスの維持

**ユーザーストーリー:** ユーザーとして、新しいインデックスが追加されてもアプリの起動時間が大幅に増加しないようにしたい。

#### 受入基準

1. WHEN インデックスを生成する時 THEN SystemはloadAllDataOnce()で1回のみ実行する
2. WHEN インデックス生成中の時 THEN Systemは進捗コールバックで「インデックス生成中...」を表示する
3. WHEN 全てのインデックスが生成された時 THEN Systemはメモリ使用量の増加を最小限に抑える
4. WHEN 検索を実行する時 THEN Systemはインデックスを使用して検索速度を向上させる

### 要件8: 後方互換性の維持

**ユーザーストーリー:** 開発者として、既存のAPIを変更せずに新しいインデックスを追加したい。

#### 受入基準

1. WHEN 新しいインデックスを追加する時 THEN Systemは既存のAPI（loadBusStops()、loadTimetable()等）を変更しない
2. WHEN 新しいデータを公開する時 THEN SystemはDataLoaderの追加プロパティとして公開する
3. WHEN 既存コードを実行する時 THEN Systemは新しいインデックスの影響を受けない
4. WHEN 既存のテストを実行する時 THEN Systemは全てのテストが成功する

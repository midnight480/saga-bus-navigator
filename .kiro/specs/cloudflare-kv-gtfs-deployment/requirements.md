# 要件定義書

## はじめに

本ドキュメントは、佐賀バスナビゲーターにおけるCloudflare KVを使用したGTFSデータ管理機能の要件を定義します。現在、GTFSデータはZIPファイルとしてGit管理され、Pages Functionで直接読み込まれていますが、Pages FunctionのCPU時間制限（50ms）により大きなGTFSファイルの処理が困難になっています。本機能により、GTFSファイルをGit管理しつつCloudflare KVで高速アクセスを実現し、運用性とパフォーマンスを向上させます。

## 用語集

- **GTFS**: General Transit Feed Specification。公共交通機関の時刻表データの標準形式
- **Cloudflare_KV**: Cloudflareが提供するグローバル分散型Key-Valueストレージ
- **Pages_Function**: Cloudflare Pagesで実行されるサーバーサイド関数
- **Deploy_Hook**: デプロイ完了時に実行されるWebhookまたはワークフロー
- **DataLoader**: GTFSデータを読み込み、アプリケーション形式に変換するモジュール
- **Version_Pointer**: KVに保存された現在有効なデータバージョンを指すキー
- **Rollback**: 以前のバージョンのデータに戻す操作

## 要件

### 要件1: GTFS前処理とJSON変換

**ユーザーストーリー:** 開発者として、GTFSデータをKVに保存する前に効率的な形式に変換したい。これにより、KVからの読み込みとパースが高速化される。

#### 受入基準

1. WHEN GTFS ZIPファイルが提供される THEN システムはZIPファイルを解凍してGTFS標準ファイル（stops.txt、stop_times.txt、routes.txt、trips.txt、calendar.txt、agency.txt、fare_attributes.txt）を抽出する
2. WHEN GTFSファイルが抽出される THEN システムは各ファイルをCSVとしてパースし、JSON形式に変換する
3. WHEN stop_timesデータのサイズが25MBを超える THEN システムは自動的にデータを複数のチャンクに分割する
4. WHEN JSON変換が完了する THEN システムは変換されたJSONファイルを出力ディレクトリに保存する
5. WHEN 変換処理中にエラーが発生する THEN システムは詳細なエラーメッセージをログに出力し、処理を中断する

### 要件2: KVへのデータアップロード

**ユーザーストーリー:** 開発者として、変換されたGTFSデータをバージョン管理しながらKVに保存したい。これにより、複数バージョンの保持とロールバックが可能になる。

#### 受入基準

1. WHEN JSONファイルが提供される THEN システムはタイムスタンプベースのバージョン番号（YYYYMMDDHHmmss形式）を生成する
2. WHEN バージョン番号が生成される THEN システムは各GTFSテーブルを`gtfs:v{version}:{table_name}`形式のキーでKVに保存する
3. WHEN stop_timesデータが分割されている THEN システムは各チャンクを`gtfs:v{version}:stop_times:{chunk_index}`形式のキーで保存する
4. WHEN 全てのデータがKVに保存される THEN システムは`gtfs:current_version`キーに新しいバージョン番号を保存する
5. WHEN KVへの保存が完了する THEN システムは保存されたキーのリストとバージョン情報をログに出力する
6. IF KVへの保存中にエラーが発生する THEN システムは部分的に保存されたデータをクリーンアップし、エラーを報告する

### 要件3: Pagesデプロイ後の自動KVアップロード

**ユーザーストーリー:** 運用者として、GTFSファイルをGitにプッシュしたら、Pagesデプロイ完了後に自動的にKVへのアップロードが実行されてほしい。これにより、非エンジニアでもCloudflareの管理画面のみで運用でき、GitHub周りの設定が不要になる。

#### 受入基準

1. WHEN Cloudflare Pagesへのデプロイが成功する THEN システムはCloudflare Deploy Hookを使用してKVアップロード処理を自動的にトリガーする
2. WHEN Deploy Hookが実行される THEN システムはデプロイされたdata/saga-current.zipファイルの存在を確認する
3. WHEN ZIPファイルが確認される THEN システムはGTFS前処理スクリプトを実行してJSON変換を行う
4. WHEN JSON変換が完了する THEN システムはKVアップロードスクリプトを実行してデータをKVに保存する
5. WHEN KVアップロードが完了する THEN システムは成功ステータスとバージョン情報をCloudflareのログに記録する
6. IF いずれかのステップが失敗する THEN システムは詳細なエラー情報をCloudflareのログに記録し、運用者に通知する

### 要件4: DataLoaderのKV統合

**ユーザーストーリー:** アプリケーション開発者として、DataLoaderがKVから高速にデータを読み込み、メモリキャッシュを活用してほしい。これにより、Pages FunctionのCPU時間制限内で処理が完了する。

#### 受入基準

1. WHEN DataLoaderが初期化される THEN システムは`gtfs:current_version`キーから現在のバージョン番号を取得する
2. WHEN バージョン番号が取得される THEN システムは各GTFSテーブルを`gtfs:v{version}:{table_name}`キーでKVから読み込む
3. WHEN stop_timesデータが分割されている THEN システムは全てのチャンクを並列に読み込み、結合する
4. WHEN KVからデータが読み込まれる THEN システムはデータをメモリにキャッシュし、以降のリクエストではキャッシュから返す
5. IF KVからのデータ読み込みが失敗する THEN システムはフォールバックとしてdata/saga-current.zipから直接データを読み込む
6. WHEN DataLoader.clearCache()が呼び出される THEN システムはメモリキャッシュをクリアし、次回のリクエストでKVから再読み込みする

### 要件5: バージョン管理とロールバック

**ユーザーストーリー:** 運用者として、問題が発生した場合に以前のバージョンのGTFSデータに戻せるようにしたい。また、古いバージョンは自動的に削除されてストレージコストを抑えたい。これにより、データ更新による障害から迅速に復旧でき、かつ運用コストを最適化できる。

#### 受入基準

1. WHEN 新しいバージョンがKVにアップロードされる THEN システムは以前のバージョンのデータを保持する
2. WHEN 新しいバージョンのアップロードが完了する THEN システムは2世代以前のバージョンのデータをKVから自動的に削除する
3. WHEN バージョン削除が実行される THEN システムは当世代と1世代前のバージョンのみをKVに保持する
4. WHEN ロールバックコマンドが実行される THEN システムは1世代前のバージョンの存在を確認する
5. WHEN バージョンが確認される THEN システムは`gtfs:current_version`キーを1世代前のバージョン番号に更新する
6. WHEN current_versionが更新される THEN システムは次回のDataLoader初期化時から指定されたバージョンのデータを使用する
7. WHEN バージョン一覧コマンドが実行される THEN システムはKVに保存されている全てのバージョン（最大2世代）とそのタイムスタンプを表示する

### 要件6: エラーハンドリングとフォールバック

**ユーザーストーリー:** アプリケーション利用者として、KVが利用できない場合でもアプリケーションが正常に動作してほしい。これにより、インフラ障害時でもサービスが継続できる。

#### 受入基準

1. IF KVへの接続が失敗する THEN システムはエラーをログに記録し、ZIPファイル読み込みにフォールバックする
2. IF KVからのデータ読み込みが5秒以内に完了しない THEN システムはタイムアウトし、ZIPファイル読み込みにフォールバックする
3. WHEN ZIPファイル読み込みにフォールバックする THEN システムはフォールバックが発生したことをログに記録する
4. WHEN フォールバック処理が完了する THEN システムは通常通りデータを返し、アプリケーションは正常に動作する
5. IF ZIPファイルも読み込めない THEN システムは明確なエラーメッセージをユーザーに表示し、処理を中断する

### 要件7: パフォーマンス要件

**ユーザーストーリー:** アプリケーション利用者として、バス停検索や時刻表表示が高速に行われてほしい。これにより、快適なユーザー体験が提供される。

#### 受入基準

1. WHEN DataLoaderがKVからデータを読み込む THEN システムは各テーブルの読み込みを5ms以内に完了する
2. WHEN stop_timesデータが分割されている THEN システムは全てのチャンクの読み込みを並列実行し、合計20ms以内に完了する
3. WHEN メモリキャッシュが有効な場合 THEN システムはキャッシュからのデータ取得を1ms以内に完了する
4. WHEN Pages Functionが実行される THEN システムは全体の処理時間を50ms以内に収める
5. WHEN 初回アクセス時にキャッシュが空の場合 THEN システムはKVからのデータ読み込みとキャッシュ構築を含めて50ms以内に完了する

### 要件8: セキュリティとアクセス制御

**ユーザーストーリー:** システム管理者として、KVへのアクセスが適切に制御され、不正なデータ更新が防止されてほしい。これにより、データの整合性とセキュリティが保たれる。

#### 受入基準

1. WHEN KVアップロードスクリプトが実行される THEN システムはCloudflare API Tokenの有効性を検証する
2. WHEN API Tokenが無効または期限切れの場合 THEN システムはエラーを報告し、処理を中断する
3. WHEN Deploy Hookが実行される THEN システムは環境変数からAPI Tokenを安全に取得する
4. WHEN Pages Functionが実行される THEN システムはKVへの読み取り専用アクセスのみを使用する
5. WHEN 本番環境でKVアップロードが実行される THEN システムは実行ログを記録し、監査可能にする

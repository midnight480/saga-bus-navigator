# 要件定義書

## はじめに

佐賀バスナビゲーターアプリにおいて、現在は佐賀駅から出発する方向のバス路線のみが検索・表示されており、佐賀駅へ向かう方向（帰り方向）のバス路線が検索結果に表示されない問題があります。この要件定義書では、双方向のバス路線検索と表示を実現するための要件を定義します。

## 用語集

- **System**: 佐賀バスナビゲーターアプリケーション
- **GTFS**: General Transit Feed Specification（公共交通データの標準形式）
- **direction_id**: GTFSにおける路線の方向を示すフィールド（0=往路、1=復路）
- **trip_headsign**: 便の行き先を示すフィールド
- **往路**: バス路線の行き方向（例：佐賀駅から目的地へ）
- **復路**: バス路線の帰り方向（例：目的地から佐賀駅へ）
- **乗車バス停**: ユーザーが乗車するバス停
- **降車バス停**: ユーザーが降車するバス停

## 要件

### 要件1: 双方向路線データの認識

**ユーザーストーリー:** システム管理者として、GTFSデータから往路と復路の両方向の路線情報を正しく認識したい。これにより、ユーザーがどちらの方向でも検索できるようにする。

#### 受入基準

1. WHEN GTFSデータを読み込む時 THEN Systemは各tripの`direction_id`フィールドを読み取る
2. WHEN `direction_id`が空または存在しない時 THEN Systemは`trip_headsign`フィールドから方向を推測する
3. WHEN 同じ路線で異なる`trip_headsign`を持つtripが存在する時 THEN Systemはそれらを異なる方向として認識する
4. WHEN 路線データを変換する時 THEN Systemは各時刻表エントリに方向情報を含める
5. WHEN 方向情報を判定できない時 THEN Systemはデフォルトで往路として扱う

### 要件2: バス停間の経路方向判定

**ユーザーストーリー:** ユーザーとして、乗車バス停と降車バス停を指定した時に、その経路が往路なのか復路なのかを自動的に判定してほしい。これにより、正しい方向のバスのみが検索結果に表示される。

#### 受入基準

1. WHEN ユーザーが乗車バス停と降車バス停を選択する時 THEN Systemは各tripの`stop_times`から停車順序を確認する
2. WHEN 乗車バス停の`stop_sequence`が降車バス停の`stop_sequence`より小さい時 THEN Systemはその方向のtripを検索対象とする
3. WHEN 複数の方向で同じバス停ペアが存在する時 THEN Systemは両方向のtripを検索対象とする
4. WHEN バス停間の経路が存在しない時 THEN Systemは適切なエラーメッセージを表示する

### 要件3: 双方向検索結果の表示

**ユーザーストーリー:** ユーザーとして、佐賀駅を降車バス停として設定した時にも、正しく検索結果が表示されてほしい。これにより、帰宅時の経路も簡単に検索できる。

#### 受入基準

1. WHEN ユーザーが佐賀駅を降車バス停として検索する時 THEN Systemは佐賀駅へ向かう方向のバスを検索結果に含める
2. WHEN 検索結果を表示する時 THEN Systemは各バスの行き先（`trip_headsign`）を表示する
3. WHEN 同じ時刻に複数方向のバスが存在する時 THEN Systemは両方を検索結果に表示する
4. WHEN 検索結果が0件の時 THEN Systemは「該当する便が見つかりません」というメッセージを表示する

### 要件4: 路線図の双方向表示

**ユーザーストーリー:** ユーザーとして、地図上で往路と復路の両方の路線を確認したい。これにより、バスがどの方向に進むのかを視覚的に理解できる。

#### 受入基準

1. WHEN 路線を地図上に表示する時 THEN Systemは往路と復路の両方のバス停を表示する
2. WHEN 往路と復路で異なるバス停を経由する時 THEN Systemは両方のバス停を地図上に表示する
3. WHEN ユーザーが特定の方向を選択した時 THEN Systemはその方向のバス停のみをハイライト表示する
4. WHEN 路線図を表示する時 THEN Systemは往路と復路を視覚的に区別できるようにする（例：色分け、矢印表示）

### 要件5: 後方互換性の維持

**ユーザーストーリー:** 開発者として、既存の機能を壊さずに双方向対応を追加したい。

#### 受入基準

1. WHEN 新しい方向対応機能を追加する時 THEN Systemは既存のデータ構造を維持する
2. WHEN 既存のAPIを呼び出す時 THEN Systemは従来通りの動作を保証する
3. WHEN 方向情報が利用できない古いデータを読み込む時 THEN Systemは従来の動作にフォールバックする
4. WHEN 既存のテストを実行する時 THEN Systemは全てのテストが成功する

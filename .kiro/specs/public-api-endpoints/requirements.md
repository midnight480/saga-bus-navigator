# 要件定義書

## はじめに

佐賀バスナビゲーターアプリに、外部から利用可能なREST APIエンドポイントを追加します。これにより、バス停情報・時刻表・経路検索機能をプログラマティックに利用できるようになります。

## 用語集

- **API**: Application Programming Interface - プログラムから利用可能なインターフェース
- **REST API**: REpresentational State Transfer API - HTTPプロトコルを使用したAPI設計スタイル
- **エンドポイント**: APIにアクセスするためのURL
- **GTFS**: General Transit Feed Specification - 公共交通機関の時刻表と地理情報を記述する標準形式
- **Cloudflare Pages Functions**: Cloudflare Pagesで提供されるサーバーレス関数実行環境
- **CORS**: Cross-Origin Resource Sharing - 異なるオリジンからのリソースアクセスを制御する仕組み
- **DataLoader**: GTFSデータを読み込み、変換、キャッシュするモジュール
- **SearchController**: 直通便の検索、フィルタリング、ソートを担当するコントローラー
- **NTP**: Network Time Protocol - ネットワーク経由で正確な時刻を取得するプロトコル
- **始発**: その日の最初に運行されるバス
- **終電**: その日の最後に運行されるバス（バスの場合は「終バス」とも呼ばれる）

## 要件

### 要件1: バス停検索API

**ユーザーストーリー**: 外部アプリケーション開発者として、バス停名で検索し、次の発車時刻と路線情報を取得したい。

#### 受入基準

1. WHEN APIエンドポイント `/api/stops/search` にGETリクエストが送信される THEN システムはバス停検索を実行する
2. WHEN クエリパラメータ `q` にバス停名が指定される THEN システムは部分一致でバス停を検索する
3. WHEN 検索結果が存在する THEN システムは現在時刻以降の次の発車時刻を含むバス停情報を返す
4. WHEN 検索結果が存在しない THEN システムは空の配列を返す
5. WHEN 認証情報が提供されない THEN システムはリクエストを受け付ける（認証不要）

### 要件2: 経路検索API

**ユーザーストーリー**: 外部アプリケーション開発者として、始点と終点を指定して経路を検索し、時刻表情報を取得したい。

#### 受入基準

1. WHEN APIエンドポイント `/api/routes/search` にGETリクエストが送信される THEN システムは経路検索を実行する
2. WHEN クエリパラメータ `from` と `to` にバス停名が指定される THEN システムは直通便を検索する
3. WHEN 検索結果が存在する THEN システムは最大5件の時刻表結果を返す
4. WHEN 検索結果に出発時刻、到着時刻、乗車時間、運賃、経由バス停が含まれる THEN システムはこれらの情報をJSON形式で返す
5. WHEN 認証情報が提供されない THEN システムはリクエストを受け付ける（認証不要）

### 要件3: 始発・終電検索API

**ユーザーストーリー**: 外部アプリケーション開発者として、指定バス停の始発・終電情報を取得したい。

#### 受入基準

1. WHEN APIエンドポイント `/api/stops/first-last` にGETリクエストが送信される THEN システムは始発・終電検索を実行する
2. WHEN クエリパラメータ `stop` にバス停名が指定される THEN システムは指定バス停の始発・終電を検索する
3. WHEN 始発情報が存在する THEN システムは始発の時刻、行先、所要時間を返す
4. WHEN 終電情報が存在する THEN システムは終電の時刻、行先、所要時間を返す
5. WHEN 認証情報が提供されない THEN システムはリクエストを受け付ける（認証不要）

### 要件4: APIドキュメント表示

**ユーザーストーリー**: アプリケーション利用者として、APIの使い方を確認したい。

#### 受入基準

1. WHEN フッターに「API」タブが追加される THEN システムはAPIドキュメントへのリンクを表示する
2. WHEN 「API」タブがクリックされる THEN システムはAPIドキュメントモーダルを表示する
3. WHEN APIドキュメントが表示される THEN システムは各エンドポイントの使い方、リクエスト例、レスポンス例を表示する
4. WHEN APIドキュメントが表示される THEN システムはエンドポイントのURL、HTTPメソッド、パラメータ、レスポンス形式を明示する
5. WHEN APIドキュメントが表示される THEN システムは実際に動作するサンプルコードを提供する

### 要件5: CORS対応

**ユーザーストーリー**: 外部アプリケーション開発者として、異なるドメインからAPIを利用したい。

#### 受入基準

1. WHEN APIエンドポイントにクロスオリジンリクエストが送信される THEN システムは適切なCORSヘッダーを返す
2. WHEN プリフライトリクエスト（OPTIONS）が送信される THEN システムは許可されたメソッドとヘッダーを返す
3. WHEN APIレスポンスが返される THEN システムは `Access-Control-Allow-Origin: *` ヘッダーを含める
4. WHEN APIレスポンスが返される THEN システムは `Access-Control-Allow-Methods: GET, OPTIONS` ヘッダーを含める
5. WHEN APIレスポンスが返される THEN システムは `Access-Control-Allow-Headers: Content-Type` ヘッダーを含める

### 要件6: エラーハンドリング

**ユーザーストーリー**: 外部アプリケーション開発者として、エラーが発生した場合に適切なエラーメッセージを受け取りたい。

#### 受入基準

1. WHEN 必須パラメータが不足している THEN システムは400 Bad Requestステータスコードとエラーメッセージを返す
2. WHEN 指定されたバス停が存在しない THEN システムは404 Not Foundステータスコードとエラーメッセージを返す
3. WHEN サーバー内部でエラーが発生する THEN システムは500 Internal Server Errorステータスコードとエラーメッセージを返す
4. WHEN エラーレスポンスが返される THEN システムはJSON形式で `{ "error": "エラーメッセージ" }` を返す
5. WHEN タイムアウトが発生する THEN システムは504 Gateway Timeoutステータスコードとエラーメッセージを返す

### 要件7: レスポンスタイム

**ユーザーストーリー**: 外部アプリケーション開発者として、APIが高速に応答することを期待する。

#### 受入基準

1. WHEN バス停検索APIが呼び出される THEN システムは500ms以内にレスポンスを返す
2. WHEN 経路検索APIが呼び出される THEN システムは500ms以内にレスポンスを返す
3. WHEN 始発・終電検索APIが呼び出される THEN システムは500ms以内にレスポンスを返す
4. WHEN GTFSデータがキャッシュされている THEN システムはキャッシュから高速にデータを取得する
5. WHEN レスポンスタイムが500msを超える THEN システムはログに警告を記録する

### 要件8: データ整合性

**ユーザーストーリー**: 外部アプリケーション開発者として、APIから返されるデータが正確であることを期待する。

#### 受入基準

1. WHEN APIがバス停情報を返す THEN システムは既存のDataLoaderと同じGTFSデータを使用する
2. WHEN APIが時刻表情報を返す THEN システムは既存のSearchControllerと同じ検索ロジックを使用する
3. WHEN APIが運賃情報を返す THEN システムは既存のFareCalculatorと同じ運賃計算ロジックを使用する
4. WHEN APIが現在時刻を使用する THEN システムはNTPから取得した正確な時刻を使用する
5. WHEN APIがレスポンスを返す THEN システムはWebアプリケーションと同じデータ形式を使用する

### 要件9: レート制限（将来実装）

**ユーザーストーリー**: システム管理者として、APIの過度な利用を防ぎたい。

#### 受入基準

1. WHEN レート制限機能が実装される THEN システムはIPアドレスごとにリクエスト数を制限する
2. WHEN レート制限を超えるリクエストが送信される THEN システムは429 Too Many Requestsステータスコードを返す
3. WHEN レート制限情報が必要な場合 THEN システムは `X-RateLimit-Limit` ヘッダーで制限値を返す
4. WHEN レート制限情報が必要な場合 THEN システムは `X-RateLimit-Remaining` ヘッダーで残りリクエスト数を返す
5. WHEN レート制限情報が必要な場合 THEN システムは `X-RateLimit-Reset` ヘッダーでリセット時刻を返す

### 要件10: APIバージョニング（将来実装）

**ユーザーストーリー**: 外部アプリケーション開発者として、APIの後方互換性を維持したい。

#### 受入基準

1. WHEN APIバージョニングが実装される THEN システムはURLパスに `/api/v1/` のようなバージョン番号を含める
2. WHEN 新しいAPIバージョンがリリースされる THEN システムは古いバージョンを一定期間サポートする
3. WHEN 非推奨のAPIバージョンが使用される THEN システムは `Deprecation` ヘッダーで警告を返す
4. WHEN APIバージョンが指定されない THEN システムは最新バージョンを使用する
5. WHEN APIドキュメントが表示される THEN システムは各バージョンの違いを明示する

### 要件11: バス停グループ検索

**ユーザーストーリー**: 外部アプリケーション開発者として、複数ののりばを持つバス停（例：佐賀駅バスセンター）を一括で検索したい。

#### 受入基準

1. WHEN バス停検索APIで親バス停名（例：「佐賀駅バスセンター」）が指定される THEN システムは関連する全てののりばの情報を統合して返す
2. WHEN バス停グループが返される THEN システムは各のりばの次の発車時刻を時刻順にソートして返す
3. WHEN バス停グループが返される THEN システムはグループタイプ（`type: "group"`）を明示する
4. WHEN バス停グループが返される THEN システムは個別ののりば情報（`platforms`配列）を含める
5. WHEN バス停グループが返される THEN システムは通常のバス停検索結果と同じレスポンス形式を維持する
